# 可验证每日数据契约实施计划

本文档用于 review `daily-free-a-share-prediction` 项目在实施前如何生成一份可验证的每日数据契约报告。当前阶段只定义实施方法、输出报告和验收标准；如果 review 没有问题，再进入接口探测和契约生成实施。

## 背景和目标

用户目标不是先假设需要哪些字段，而是先测试 AKShare 今日实际能获取的最大数据集合，再对今日可用的数据逐项向历史日期回溯，直到找到每个数据项的最早可获取交易日。

最终交付物是一份报告，回答三个问题：

1. 今日通过 AKShare 接口到底能获取哪些 A 股相关数据。
2. 这些今日可获取数据中，哪些可以对历史交易日继续获取。
3. 在“不允许空数据、不允许接口不可调用、不允许占位补齐”的前提下，哪些数据可以进入每日数据契约，以及对应最早交易日是哪一天。

## 核心原则

- 契约来自接口实测，不来自文档推断。
- 今日不可调用、今日返回空、字段无法解析、字段不稳定的接口，不进入候选契约。
- 历史探测只对今日通过的接口执行，避免为已经不可用的数据浪费设计成本。
- 每日数据契约只包含可通过接口获取到的非空数据，不用空数组、空行、默认值或占位值补齐。
- 最终报告必须保留排除原因；排除结果和纳入结果同等重要。
- 数据越多，统一历史起点通常越晚；历史越长，能进入契约的数据通常越少。报告必须显式展示这个取舍。

## 待实施工作 DAG

1. 候选接口清单整理
   - 输入：AKShare 当前安装版本、A 股相关接口函数、接口签名。
   - 输出：`candidate_interfaces.json`。
   - 依赖：项目 Python 环境可导入 AKShare。
   - 触碰模块：后续实施时新增 `scripts/` 和 `reports/`。
   - 风险：接口数量过多导致首轮探测不稳定。
   - 验证：每个候选接口都有函数名、参数策略、数据频率初判和探测方式。

2. 今日最大数据探测
   - 输入：候选接口清单、基准日期，默认使用实施当天。
   - 输出：`today-capability-report.json` 和可读 Markdown 摘要。
   - 依赖：候选接口清单。
   - 触碰模块：探测脚本和报告输出目录。
   - 风险：代理、上游限流、非交易日导致结果不稳定。
   - 验证：每个接口记录 `call_status`、`row_count`、`columns`、`sample_rows`、`failure_reason`。

3. 今日候选契约冻结
   - 输入：今日最大数据探测结果。
   - 输出：`today-contract-candidates.json`。
   - 依赖：今日探测完成。
   - 触碰模块：报告生成逻辑。
   - 风险：把事件型数据误判为每日必有数据。
   - 验证：只有今日调用成功、非空、字段可解析的数据进入候选池。

4. 历史最早交易日探测
   - 输入：今日候选契约、交易日回溯策略。
   - 输出：`history-floor-report.json`。
   - 依赖：今日候选契约冻结。
   - 触碰模块：历史探测脚本。
   - 风险：逐日探测耗时长；接口历史边界不连续。
   - 验证：每个候选数据项都有 `first_available_trade_date`、`last_failed_before_first`、`probe_evidence`。

5. 每日数据契约生成
   - 输入：今日候选契约、历史最早交易日探测结果。
   - 输出：`daily-data-contract.v1.json` 和 Markdown 报告。
   - 依赖：历史探测完成。
   - 触碰模块：契约生成逻辑。
   - 风险：为了字段丰富度牺牲历史长度，或者为了历史长度牺牲可用字段。
   - 验证：报告给出至少两种契约视图：`max_today_contract` 和按历史起点分层的 `historical_contract_options`。

6. 契约验收验证
   - 输入：生成的每日数据契约。
   - 输出：`contract-validation-report.json`。
   - 依赖：契约生成。
   - 触碰模块：验证脚本。
   - 风险：只验证单日，遗漏跨日字段漂移。
   - 验证：从契约起点、近期日期和随机交易日抽样，所有纳入字段都能非空获取并通过字段一致性检查。

## 今日最大数据探测规则

今日探测是第一道门槛。接口必须同时满足以下条件，才能进入下一阶段：

- 函数存在且可调用。
- 参数可以由脚本稳定构造。
- 返回结构能解析为表格或明确的结构化数据。
- 返回行数大于 0。
- 字段名、字段数量和关键字段类型可记录。
- 调用失败时能捕获异常并写入失败原因。

今日探测输出字段：

```json
{
  "interface_name": "stock_zh_a_hist",
  "probe_date": "2026-06-18",
  "call_status": "success",
  "row_count": 100,
  "columns": ["日期", "开盘", "收盘"],
  "sample_rows": [],
  "data_frequency": "unknown_until_verified",
  "contract_candidate": true,
  "failure_reason": null
}
```

排除规则：

- `call_status != success`：排除。
- `row_count <= 0`：排除。
- `columns` 为空或无法稳定解析：排除。
- 需要人工输入、验证码、登录态或付费权限：排除。
- 今日可用但只代表当前快照的数据，保留在报告中，但默认不进入历史每日契约。

## 历史最早交易日探测规则

历史探测只处理今日已通过的数据项。每个候选接口都需要回答：

- 是否支持指定历史日期或日期范围。
- 最早能返回非空结果的交易日。
- 最早可用日前一天或前一个探测窗口为何不可用。
- 历史字段是否与今日字段一致。
- 是否存在近期可用、远期不可用、或中间断档。

历史探测类型：

- `daily_range`：可以按交易日或日期范围稳定获取历史数据，是每日契约优先候选。
- `recent_only`：可以按日期获取，但历史窗口有限，只能形成较短历史契约。
- `event_window`：事件发生才有数据，不适合作为每日必有契约，可作为事件增强报告。
- `latest_snapshot`：只能代表获取时点，不进入历史每日契约。
- `unsupported_history`：今日能取到，但无法回溯历史，不进入历史契约。

历史探测策略：

- 先用粗粒度年份或月份探测缩小范围。
- 再在边界区间内按交易日精确查找第一天非空结果。
- 对非交易日不计为失败，但必须记录跳过原因。
- 对接口报错、代理失败、上游超时与真实无数据分开记录。

## 契约生成规则

每日数据契约分成两个视图，不强行合并：

1. `max_today_contract`
   - 表示今日能通过 AKShare 获取到的最大非空数据集合。
   - 可以包含快照型和事件型数据。
   - 只用于说明今日能力边界，不承诺历史回放。

2. `historical_daily_contract`
   - 表示在同一历史日期范围内，每天都能通过接口获取到的非空数据集合。
   - 只允许纳入 `daily_range` 或经实测可连续回放的 `recent_only` 数据。
   - 必须给出 `contract_start_date`。

契约字段必须包含：

```json
{
  "contract_version": "daily_data_contract.v1",
  "generated_at": "2026-06-18T00:00:00+08:00",
  "akshare_version": "1.18.64",
  "contract_start_date": "YYYY-MM-DD",
  "interfaces": [],
  "excluded_interfaces": [],
  "validation_status": "passed"
}
```

每个纳入接口必须包含：

```json
{
  "interface_name": "stock_zh_a_hist",
  "data_frequency": "daily_range",
  "first_available_trade_date": "YYYY-MM-DD",
  "required_parameters": {},
  "columns": [],
  "required_non_empty": true,
  "field_stability_status": "passed",
  "evidence_refs": []
}
```

## 最终报告结构

最终 Markdown 报告建议命名为 `reports/daily-data-contract-report.md`，内容包括：

1. 执行摘要
   - 今日探测日期。
   - AKShare 版本。
   - 总候选接口数量。
   - 今日可用接口数量。
   - 历史可回放接口数量。
   - 推荐每日契约起始日。

2. 今日最大可获取数据
   - 可用接口表。
   - 每个接口行数、字段、样例摘要。
   - 今日不可用接口及原因。

3. 历史最早可获取日
   - 每个今日可用接口的历史分类。
   - 每个接口的最早非空交易日。
   - 字段一致性检查结果。

4. 推荐每日数据契约
   - 纳入接口。
   - 纳入字段。
   - 起始交易日。
   - 为什么这些数据满足“每日可获取、非空、可验证”。

5. 排除清单
   - 接口不可调用。
   - 今日为空。
   - 历史不可回放。
   - 事件型自然为空。
   - 快照型无法代表历史。

6. 验证证据
   - 探测命令。
   - 样例日期。
   - 成功和失败样例。
   - 报告生成时间。

## 验收标准

实施完成后，必须同时满足：

- 能生成今日最大能力报告。
- 能生成历史最早可获取日报告。
- 能生成每日数据契约 JSON。
- 能生成面向 review 的 Markdown 报告。
- 所有纳入契约的数据项都有非空接口调用证据。
- 所有排除的数据项都有明确排除原因。
- 报告明确区分 `max_today_contract` 和 `historical_daily_contract`。
- 报告不得把未验证数据写成已验证事实。

## 当前已知约束

- 当前项目 `.venv` 暂未安装 AKShare；实施时需要先补齐依赖。
- 当前工作区不是 git 仓库，无法用 `git status` 检查变更边界。
- 之前本机探测显示全局 Python 可导入 AKShare，但项目实施应以本项目 `.venv` 为准。
- 历史接口能力需要实测确认，不能只依赖 AKShare 文档或过去经验。

## 不做事项

- 不在本阶段生成预测结果。
- 不在本阶段选择股票池或交易策略。
- 不用空数据补齐契约。
- 不把事件型数据强行定义为每日必有数据。
- 不把快照型数据误写成历史日数据。
- 不引入 AKShare 之外的数据源，除非用户后续明确要求。
