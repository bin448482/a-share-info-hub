从数据管道到 AI Agent 编排：构建一个 A 股研究信息中枢

一个关于架构决策、AI Agent 编排，以及如何用确定性代码约束 LLM 边界的项目复盘。

动机

每天 A 股收盘后，我需要回答一系列研究问题：大盘今天怎么样？涨停池里有没有值得关注的结构性变化？龙虎榜资金在往哪儿流？宏观面和主要投行观点对市场情绪有什么影响？

手动做这件事有两个问题：重复性劳动太多（15+ 个公开数据接口，逐一手工查询、对字段、做标准化），以及纯脚本缺少分析上下文（能拿到数据，但数据之间的关联和叙事需要人力判断）。

传统方案是写一个爬虫脚本定时拉数据。但我想要的不只是数据 dump — 我想要一个能自动生成结构化研究报告的系统，同时确保每一条结论都可以追溯到数据来源，而且绝对不会越界给出交易建议。

于是有了 a-share-info-hub。

架构全景

整个系统的核心设计原则是：Python 做确定性工作，AI Agent 做需要判断力的编排和分析。

第一层：数据采集（Python 确定性管道）

每天收盘后，一条 CLI 命令 python -m a_share_info_hub daily-update 触发采集链：

  • 先验证目标日期是否 A 股交易日（通过 AKShare 交易日历）
  • 如果是交易日，并行拉取 14 个公开接口：全市场个股快照（~5500 只）、6 个涨跌停池、3 个龙虎榜维度、沪深交易所概况、行业和概念板块
  • 每个接口独立记录状态（passed / partial / failed / schema_changed），失败不阻塞其他接口
  • 原始响应以 JSON 完整保留，标准化后写入 Parquet + DuckDB

这里有一个重要的设计：不假设 API 字段永远不变。国内免费数据接口经常调整字段名，所以我实现了字段别名解析（find_column() 基于已知别名元组做模糊匹配），以及 SchemaChangedError 异常类，让模式漂移成为可观察事件而非静默故障。

第二层：证据编译（Python 业务逻辑）

数据采集完成后，daily-review 子命令将分散的 Parquet 表和接口状态编译为一个版本化的 evidence packet（review-context.json）：

  • 明确列出 allowed_sections（哪些分析板块有数据支撑）
  • 明确列出 blocked_sections（哪些数据源挂了，对应的分析板块被冻结）
  • 明确列出 forbidden_claims（LLM 不能自己编造的结论）
  • 硬编码 analysis_mode: research_only 和 not_investment_advice: true

这不是对 LLM 的软约束，而是代码层的硬约束。LLM 的输入被严格限定在 evidence packet 内，不允许引入外部知识。

第三层：Agent 编排（AI 判断力层）

复盘流程由两个 Claude Code skill 驱动：

  • a-share-daily-review：主导完整工作流 — 生成 evidence packet → 传递给 LLM 写 sections JSON → Pydantic 结构校验 → 业务规则扫描 → 渲染用户可读 HTML
  • daily-financial-briefing：负责外部背景 — 从公开来源搜集 US Macro（CPI、FOMC、国债收益率）和主要投行观点，带来源引用和发表日期

当需要外部背景时，编排模式是 6 路并行子 Agent 的扇出/扇入拓扑：

父 Agent（daily-review）
  ├── 子 Agent 1：大盘定性 + external background
  ├── 子 Agent 2：大盘结构 + external background
  ├── 子 Agent 3：市场宽度 + external background
  ├── 子 Agent 4：情绪与事件 + external background
  ├── 子 Agent 5：板块与结构 + external background
  └── 子 Agent 6：风险观察 + external background
       ↓
  确定性代码合并为 external_background_fusion.v1

每个子 Agent 使用 $daily-financial-briefing skill 独立搜索自己负责的主题。合并过程由确定性 Python 代码完成，不交给 LLM — 因为 LLM 在汇总结构化数据时容易出错，而代码不会。

五个关键设计决策

1. research-only 不是提示词，是架构

很多 AI 应用把"不提供医疗建议/法律建议/投资建议"写成 prompt 里的免责声明。用户换个问法就绕过去了。

在这个项目里，交易行为过滤是多层的、硬编码的：

  • Prompt 层：LLM 被明确禁止使用买入/卖出/加仓/止损等语言
  • Python 层：contains_trade_action_request() 在用户输入进入任何数据分析之前就扫描交易模式并直接拒绝；enforce_user_report_language_boundary() 扫描最终 HTML，确保任何内部诊断术语（如 data_status: partial、blocked_sections）不会泄露给普通用户
  • 业务规则层：validate_report_business_rules() 验证 LLM 输出没有对被 blocked 的数据源做出正面结论

决定：安全约束放在代码里，不在 prompt 里祈祷。
代价：维护四套禁止词列表（FORBIDDEN_OUTPUT_TERMS、TRADE_ACTION_PATTERNS、BLOCKED_SECTION_FORBIDDEN_TERMS、LOW_INFORMATION_EXTERNAL_PHRASES），但换来的是确定性的安全保证。

2. 两层验证架构

LLM 输出是不可靠的—可能缺少字段、可能编造数据、可能违反输出契约。

项目采用两层门禁：

  1. Pydantic 结构验证：model_validator 和 field_validator 强制执行类型、必填字段、禁止额外字段和日期格式
  2. 业务规则验证：扫描禁止词、blocked section 越界、虚假完整性声明（"本报告覆盖了全部数据" — 当有数据源 failed 时这句话就是假的）

任何一层未通过，渲染直接中断并返回具体失败原因。

这让我可以在没有 LLM 的情况下测试整个流水线 — 系统提供了 deterministic 渲染模式，接收相同的 context 产出相同结构的 sections，使得 LLM 异常时能快速定位问题在 context 还是 LLM。

3. 数据缺口不作静默失败

很多数据管道把"这个接口挂了"当作一条 error log 然后跳过。在这个项目里，数据缺口是输出的一部分：

  • 每个 API 的状态独立追踪（passed / partial / failed / skipped / schema_changed / empty）
  • 状态通过 apply_status_rules() 编译为整体画像
  • 如果一个增强接口挂了，主表 passed 时整体是 partial 而非 failed
  • LLM 被指示：如果某个板块的数据不可用，描述数据缺口，不要猜测

这套状态机确保：你可以一眼看出哪些数据是可靠的，哪些是缺失的，以及缺失对结论有什么影响。

4. 采集与分析完全分离

采集脚本（collect_daily_snapshot.py，1365 行）不接触 LLM。
LLM 分析流程不接触 API。

两个世界通过文件系统通信：interface-status.json → review-context.json → llm-review-sections.json → HTML。

这意味着：

  • 采集层的稳定性可以在无 LLM 的情况下独立测试
  • LLM 输出的质量可以用 mock 数据独立评估
  • 如果 AKShare API 挂了，不影响已有数据的复盘分析
  • 如果 LLM 输出有问题，不会污染数据采集

5. 契约式设计

所有组件间的 JSON 契约都带有 Literal 锁定的 schema_version（如 daily_review_context.v1、external_background_fusion.v1）。

字段演进时，上下游通过 version 字符串识别格式变化 — 不会出现"字段悄悄改了，下游误读了 3 天才发现"的情况。

几点反思

1. Prompt 约束是脆弱的，代码约束是可靠的。

在写这个项目之前，我会把所有行为期望写在 prompt 里。现在我的做法是：prompt 负责"怎么表达"（写作风格、角色语气），代码负责"不能说什么"（禁用词、越界检测、安全边界）。prompt 里的约束是给 LLM 的建议，代码里的约束才是真正的 gate。

2. 为离线测试做好铺垫。

deterministic 渲染模式和基于 fixture 的 Promptfoo 评测是这个项目最有价值的工程投资之一。它们让我可以在不烧 token 的情况下验证核心逻辑是否正确 — 每次改 prompt 或 validator 之后，跑一遍 npm run test:eval，几分钟就知道有没有退化。

3. 把 LLM 当受控文本生成组件，而不是流程编排器。

LLM 擅长生成有洞察力的文本，但在结构化数据汇总和确定性逻辑链路中不够可靠。我的选择是：LLM 负责生成 sections 的自然语言内容，Python 负责合并、校验、渲染。结果是：我得到了 LLM 的分析深度，也得到了软件工程的确定性保证。

4. 契约先于实现。

在写任何采集代码之前，先定义了数据模式版本和输出契约。在写任何 LLM prompt 之前，先定义了什么算"pass"。这种"先定标准再写代码"的方式，在 LLM 开发中比传统软件工程更重要 — 因为 LLM 的输出质量是统计性的，你需要一个确定性的标准来判断它是否达标。
