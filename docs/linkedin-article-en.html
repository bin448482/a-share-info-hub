From Data Pipeline to AI Agent Orchestration: Building an A-Share Research Info Hub

A project postmortem on architecture decisions, AI agent orchestration, and why I chose to constrain LLMs with deterministic code instead of prompt engineering.

The Motivation

Every day after the A-share market closes, I found myself asking the same set of research questions: How did the market do? Any structural changes in the limit-up pools? Where is institutional money flowing on the Dragon Tiger List? What does the macro landscape and major bank commentary suggest about near-term sentiment?

Doing this manually meant highly repetitive work (15+ public data interfaces, each requiring manual query, field mapping, and normalization) and insufficient analytical context (raw data dumps don't tell stories — the connections between data points require synthesis).

A traditional approach would be a cron-triggered scraper. But I didn't just want a data dump. I wanted a system that automatically generates structured research reports — where every conclusion is traceable to a data source, and where the boundary between "research" and "trading advice" is enforced in code, not in wishful prompting.

That's the origin story of a-share-info-hub.

Architecture at a Glance

The system is built on a single principle: Python handles deterministic work; AI agents handle judgment, synthesis, and orchestration.

Layer 1: Data Collection (Deterministic Python Pipeline)

One CLI command — python -m a_share_info_hub daily-update — triggers the collection chain:

  • Verify the target date is an A-share trading day (via AKShare's trading calendar)
  • If yes, pull 14 public interfaces in parallel: full-market snapshot (~5,500 stocks), 6 limit-up/down pools, 3 Dragon Tiger List dimensions, SSE/SZSE exchange summaries, industry and concept board snapshots
  • Each interface independently tracks its status (passed / partial / failed / schema_changed) — one failure never blocks others
  • Raw responses are preserved as JSON; normalized data goes to Parquet + DuckDB

One detail worth highlighting: I don't assume API fields are stable. Chinese free data interfaces frequently adjust column names and shapes. So the code has a SchemaChangedError exception class and a find_column() function that performs fuzzy matching against known alias tuples — making schema drift an observable event, not a silent failure.

Layer 2: Evidence Compilation (Python Business Logic)

After collection, the daily-review subcommand compiles the scattered Parquet tables and interface states into a single, versioned evidence packet (review-context.json):

  • Lists allowed_sections — which analytical sections have data backing
  • Lists blocked_sections — which data sources failed, freezing their corresponding report sections
  • Lists forbidden_claims — conclusions the LLM must not fabricate
  • Hardcodes analysis_mode: research_only and not_investment_advice: true

This is not a soft prompt constraint. It's a hard code constraint — the LLM's input is strictly scoped to the evidence packet. No external knowledge allowed.

Layer 3: Agent Orchestration (The Judgment Layer)

The review workflow is driven by two Claude Code skills:

  • a-share-daily-review: Drives the full pipeline — generates the evidence packet → sends it to the LLM for section writing → Pydantic validation → business rule scanning → renders user-readable HTML
  • daily-financial-briefing: Handles external context — gathers US Macro data (CPI, FOMC, Treasury yields) and major investment bank views from public sources, with full citations and publication dates

When external background is needed, the orchestration pattern is a 6-way parallel fan-out / fan-in topology:

Parent Agent (daily-review)
  ├── Sub-agent 1: Market Overview Assessment + external background
  ├── Sub-agent 2: Market Structure + external background
  ├── Sub-agent 3: Market Breadth + external background
  ├── Sub-agent 4: Sentiment & Events + external background
  ├── Sub-agent 5: Board & Structure + external background
  └── Sub-agent 6: Risk Observations + external background
       ↓
  Deterministic code merges into external_background_fusion.v1

Each sub-agent uses the $daily-financial-briefing skill to independently search its assigned topic. The final merge is performed by deterministic Python code — not by an LLM — because LLMs are unreliable at structured data aggregation, and code is not.

Five Design Decisions Worth Discussing

1. research-only Is an Architecture, Not a Prompt

Many AI applications handle safety boundaries with a prompt disclaimer: "Do not provide financial advice." Users can jailbreak that with a rephrase.

In this project, the safety boundary is multi-layered and hardcoded:

  • Prompt layer: The LLM is explicitly forbidden from using buy/sell/position/stop-loss language
  • Python layer: contains_trade_action_request() scans user input for trading patterns and rejects the request before any data analysis occurs; enforce_user_report_language_boundary() scans the final HTML to ensure internal diagnostics (like data_status: partial or blocked_sections) never leak to non-technical readers
  • Business rules layer: validate_report_business_rules() verifies the LLM hasn't drawn positive conclusions about blocked data sources

Decision: Enforce safety in code. Don't pray that prompts will hold.
Trade-off: Maintaining four separate forbidden-term lists (FORBIDDEN_OUTPUT_TERMS, TRADE_ACTION_PATTERNS, BLOCKED_SECTION_FORBIDDEN_TERMS, LOW_INFORMATION_EXTERNAL_PHRASES) — but the payoff is deterministic safety guarantees.

2. Dual-Layer Validation Architecture

LLM output is unreliable — it can miss fields, hallucinate data, or violate output contracts.

The system uses a two-layer gate:

  1. Pydantic structural validation: model_validator and field_validator enforce types, required fields, forbidden extra fields, and date formats
  2. Business rule validation: Scans for forbidden terms, blocked-section boundary violations, and false completeness claims ("This report covers all data" is a lie when some data sources have failed)

If either layer fails, rendering is blocked and the exact failure reason is returned.

This also means I can test the entire pipeline without an LLM — a deterministic render mode accepts the same context and produces the same section structure, enabling fast fault isolation with zero token cost.

3. Data Gaps as First-Class Citizens

Many data pipelines treat "this API is down" as an error log entry and move on. In this project, data gaps are part of the output:

  • Each API's status is independently tracked (passed / partial / failed / skipped / schema_changed / empty)
  • A state machine (apply_status_rules()) compiles them into an overall health picture
  • If an enhancement interface fails while the main table passes, overall status is partial, not failed
  • The LLM is instructed: if data for a section is unavailable, describe the gap, don't guess

This state machine ensures you can see at a glance which data is reliable, which is missing, and what impact the gaps have on conclusions.

4. Collection and Analysis Are Completely Decoupled

The collection script (collect_daily_snapshot.py, 1,365 lines) never touches an LLM.
The LLM analysis flow never touches an API.

They communicate via the filesystem: interface-status.json → review-context.json → llm-review-sections.json → HTML.

What this buys:

  • Collection stability can be tested independently with zero LLM involvement
  • LLM output quality can be evaluated with mock data
  • If AKShare APIs go down, review analysis on existing data still works
  • If LLM output is broken, the data pipeline is unharmed

5. Contract-First Design

Every inter-component JSON contract carries a Literal-locked schema_version (e.g., daily_review_context.v1, external_background_fusion.v1).

When fields evolve, consumers identify the format change through the version string. No more "field silently changed and downstream misread it for three days" situations.

Lessons Learned

1. Prompt constraints are fragile. Code constraints are reliable.

Before this project, I would have put all behavioral expectations in prompts. Now my practice is: prompts control how to express (writing style, role tone); code controls what cannot be said (forbidden terms, boundary violations, safety gates). Prompt constraints are suggestions to the LLM. Code constraints are actual gates.

2. Invest early in offline testing infrastructure.

The deterministic render mode and fixture-based Promptfoo evaluation were among the highest-ROI engineering investments in this project. They let me verify core logic without burning tokens — after every prompt or validator change, running npm run test:eval tells me in minutes whether anything regressed.

3. Treat the LLM as a constrained text-generation component, not a flow orchestrator.

LLMs excel at generating insightful text but are unreliable for structured data aggregation and deterministic logic chains. My choice: the LLM generates natural-language section content; Python handles merging, validation, and rendering. The result: I get the LLM's analytical depth, and I get software engineering's deterministic guarantees.

4. Contracts before implementation.

I defined data schema versions and output contracts before writing any collection code. I defined what counts as "pass" before writing any LLM prompt. This "define the standard before writing the code" approach matters even more in LLM development than in traditional software engineering — because LLM output quality is statistical, and you need a deterministic standard to judge whether it meets the bar.
