# 核心原则

## 编码前先思考

**不要假设。不要隐藏困惑。明确暴露权衡。**

- 明确说明你的假设。如果不确定，就提问。
- 如果存在多种理解方式，列出来；不要静默选择其中一种。
- 如果有更简单的做法，直接说明。必要时提出反对意见。
- 如果某件事不清楚，就停下来。说清楚哪里令人困惑，然后提问。

## 保持简单

**用能解决问题的最少代码。不要做推测性扩展。**

- 不添加超出请求范围的功能。
- 不为单次使用的代码抽象新层。
- 不添加未被要求的“灵活性”或“可配置性”。
- 不为不可能发生的场景添加错误处理。
- 如果你写了 200 行，而 50 行就能解决，重写并简化。

自问：“资深工程师会不会认为这过度复杂？”如果答案是会，就简化。

## 尊重当前结构

- 复用现有 helper 和分层，不要在新入口里重建逻辑。
- 匹配现有风格，即使你会用不同方式实现。
- 如果发现无关的死代码，提出来；不要删除它。

## 明确当前状态

- 行动前先读取当前工作流状态、等待点和评估上下文。
- 不要猜测业务输入。如果 JD、面试反馈或工作区输入尚未解决，就停在正确的 blocked 或 waitpoint 边界。

## 外科手术式修改

**只触碰必须触碰的内容。只清理你自己制造的杂物。**

编辑现有代码时：

- 不要“顺手改进”相邻代码、注释或格式。
- 不要重构没有坏掉的东西。

当你的改动制造了孤立代码时：

- 移除由你的改动导致未使用的 import、变量或函数。
- 不要移除预先存在的死代码，除非用户要求。

检验标准：每一行改动都应能直接追溯到用户请求。

# 执行规划与并行

- 编辑前，把非平凡请求转换为 DAG；每个工作项包含：输入、输出、依赖、触碰的文件/模块、风险和验证方式。
- 优先并行执行只读探索：代码搜索、文件读取、测试发现、配置发现和调用链检查。
- 不要并行修改同一文件、共享契约、共享状态、不稳定接口或实时业务数据。
- 实施前冻结核心契约：API 形状、schema、状态转换、工作流语义和公开入口。
- 任务较大时分批执行：并行探索 -> 串行契约 -> 并行隔离实现 -> 串行集成 -> 串行验证。
- 每个有意义的批次后，报告已完成工作、新发现的冲突或依赖，以及下一批次。
- 如果执行中出现冲突，自动从并行降级为串行，不要强行并发。
- 如果必需输入未解决，或工作流状态不明确，停在 `blocked` 或正确等待点，而不是猜测。
- 当环境共享、易波动，或已知会保留残留进程时，优先顺序执行测试。

## 目标驱动执行

**定义成功标准。循环直到验证通过。**

把任务转换为可验证目标：

- “添加校验” -> “先为无效输入写测试，再让测试通过”
- “修复 bug” -> “先写一个复现 bug 的测试，再让测试通过”
- “重构 X” -> “确保重构前后测试都通过”

对于多步骤任务，给出简短计划：

```text
1. [步骤] -> 验证：[检查]
2. [步骤] -> 验证：[检查]
3. [步骤] -> 验证：[检查]
```

强成功标准能让你独立循环推进。弱标准（例如“让它能用”）会需要不断澄清。

# 编码规范

本仓库只使用这一套编码规范。不要为不同目录、不同语言或不同工具另建一套冲突规则；如果确实需要例外，先在任务说明中写明原因。

## 目录级规则

- 每个实际存在的代码或文档目录都必须包含 `AGENTS.md` 和 `claude.md`。
- 根目录 `AGENTS.md` 保存全局工作规则。
- 子目录 `AGENTS.md` 不是重复根规则的地方，而是该目录的索引：说明目录用途、列出当前文件/子目录功能，并记录改动后需要同步更新的索引要求。
- 如果新增、删除、改名、移动该目录下的文件或子目录，必须同步更新对应目录的 `AGENTS.md` 索引。
- 每个目录的 `claude.md` 内容只允许是：

```text
@agents.md
```

- 不为了满足规范创建空目录。只有当目录承载真实代码、脚本、测试、文档或数据说明时，才创建该目录及对应的 `AGENTS.md` / `claude.md`。

## 文件注释规则

- 每个源代码文件都必须在文件顶部写明文件用途。
- 文件用途注释要说明“这个文件负责什么”，不要重复文件名。
- Markdown、JSON、YAML 等非代码文件无法自然放置代码注释时，使用标题、`description` 字段或文件开头说明表达用途。

示例：

```python
"""加载并标准化每日 A 股公开数据。"""
```

```typescript
/**
 * 生成每日研究报告的页面状态。
 */
```

## 方法注释规则

- 每个函数、方法或公开脚本入口都必须写注释说明功能。
- 注释重点说明输入、输出和业务含义；不要写“给变量赋值”这类代码本身已经表达清楚的内容。
- Python 使用 docstring；TypeScript/JavaScript 使用 JSDoc；Shell 或 PowerShell 使用紧邻方法/函数上方的行注释。

示例：

```python
def normalize_symbol(symbol: str) -> str:
    """把用户输入的股票代码转换为内部统一格式。"""
```

```typescript
/**
 * 将原始行情记录转换为报告生成所需的标准字段。
 */
function normalizeQuote(raw: QuoteRecord): NormalizedQuote {
```

## 简单优先

- 注释必须帮助读者理解用途、边界或业务含义。
- 不写空泛注释，不用注释复述显而易见的语法。
- 新增代码时先满足当前需求；不要因为“以后可能用到”添加抽象、配置或扩展点。

# 根目录索引

本节是仓库根目录索引。新增、删除、改名或移动顶层文件/目录时，必须同步更新这里的说明；子目录内部索引由对应目录的 `AGENTS.md` 维护。

## 顶层文件

- `AGENTS.md`：全局工作规则和根目录索引。
- `claude.md`：Claude/Codex 入口引用文件，内容固定为 `@agents.md`。
- `.gitignore`：本仓库忽略规则。
- `README.md`：每日快照采集链路的使用说明和验证命令。
- `requirements.txt`：Python 运行和测试依赖。
- `market.duckdb`：每日快照标准化表的本地 DuckDB 分析库，由采集脚本生成。

## 顶层目录

- `.claude/`：Claude 相关本地配置目录。
- `.codex/`：Codex 相关本地配置目录。
- `.venv/`：项目 Python 虚拟环境，不作为代码索引对象。
- `a_share_info_hub/`：仓库 CLI 包目录，日常每日更新入口为 `python -m a_share_info_hub daily-update`。
- `data/`：原始数据和标准化数据目录，详见 `data/AGENTS.md`。
- `docs/`：设计和实施文档目录，详见 `docs/AGENTS.md`。
- `logs/`：接口状态和失败日志目录，详见 `logs/AGENTS.md`。
- `reports/`：每日运行报告和可 review 产物目录，详见 `reports/AGENTS.md`。
- `scripts/`：可执行脚本目录，详见 `scripts/AGENTS.md`。
- `skills/`：仓库内 Codex/Claude skill 源文件目录，详见 `skills/AGENTS.md`。
- `eval/`：本地 prompt/skill 评测配置和 provider 目录，详见 `eval/AGENTS.md`。
- `tests/`：单元测试和 fixture 目录，详见 `tests/AGENTS.md`。
